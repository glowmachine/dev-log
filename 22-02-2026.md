## General
### Terminology
- a project is made up of a folder of containing every file and subfolder related to it
- git is a version control system that records a project's history, including copies of those files along each step of the way -- it's navigated and controlled through the command line -- to me, git is best understood as 3 spaces and 3 labels...
  - the 3 spaces: local repository, remote repository, and working directory
    1. a .git repository is a hidden folder in the project folder that contains its git system, usually referred to as the ***local repository***
    2. a ***remote repository*** is the .git repository that is somewhere else or in the cloud, like github, where you download to update your local repo or upload to backup changes or collaborate with others on an origin copy
      - "branches" are different versions of a project, usually containing relatively small changes to copies of the main branch before they're actually applied to the main branch
      - "commits" are different versions of a branch, tracking the development of things like new features and bug fixes
    3. the ***working directory*** is the space inside the project folder, it content changes based on what which commit you're looking at
  - the 3 labels: commited, staged, edited
    1. files in a commit are ***committed***, a copy of them in the state they were in at the time can be reopened by checking out that specific commit
    2. ***staged*** files are files that will be included in a commit whenever the next one is made
    3. whenever a file is ***edited***, it's automatically unstaged and need to be manually restaged to be included in the next commit, otherwise the other staged file before it, before it was edited, will be the one that's saved with the next commit
    
    - (git commits actually save the differences between files that are staged, so it's not making entire copies of files every commit -- if a file wasn't edited or no changes were staged, then the file you see with that commit will just be version of the file from the commit before it)
- merging combines the files of a different commit with the current commit into a new commit
  - the most recent commit of a branch is used if a branch name is given (typical usage) instead of a commit ID
  - retains the complete commit history of both branches, but looks messy and confusing in large projects with many branches when merged back into main
- rebasing copies commits of the current branch and make them branch off from, aka bases them off of, a different commit
  - retains the complete commit history of both branches and keeps them linear, but risks unsyncing with other contributors on the same branch since the commit IDs of the non-main branch are changed in the process
  - squashing uses interactive rebasing to collapse specific commits of the current branch into a single commit
- generally, all git commands are relative to the current commit

## Command Line / Vocabulary
- `command subcommand <option/flag/args>`
  - `command` is the first word of a line, like `git`
  - `subcommands` are the second word, like `add`, `log`, `switch`
  - `options` are parameters that change the behavior of a command or subcommand, like `-D`, `--staged`, `--set-upstream-to=`
  - `flags` are boolean options, like `-v`, `--all`, 
  - `arguments` are the variables that commands are given as input, like `helloworld.txt`, `main`, `HEAD~1`
- "branch tip" is the most recent commit of a branch

## Variables
- `HEAD`: pointer to the current commit or "location," determines the working tree (or working directory, what files are accessible)
  - `HEAD~n`: pointer to the nth previous commit in the current branch
  - "detached head": HEAD pointer points to a commit that isn't a branch tip
- commit ID: 40-digit hexidecimal values, often shortened to 7 but can be referenced using any n-digits as long as they are unambiguous
- branch names are just aliases for its branch tip
- remote repository names are just aliases to its URL, they don't have to actually match the remote repository's name

## System Setup
- `ls ~/.ssh/id_ed25519.pub`: list the `id_ed25519.pub` file if an Ed25519 SSH key
- `ssh-keygen -t ed25519`: generate an SSH key pair of type Ed25519
- `cat ~/.ssh/id_ed25519.pub`: print the public key (private key is `~/.ssh/id_ed25519` with no file extension)
- `ssh -T git@ssh.github.com`: test if SSH can connect to github
- `ssh -T -p 443 git@ssh.github.com`: test if SSH can connect to github over port 443 (typically for HTML)
  - port 22 is the default for SSH, but it'ss often blocked on public wifis
  - there may be a warning that says authenticity can't be established, just make sure the provided key fingerprint matches github's [public key fingerprints](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/githubs-ssh-key-fingerprints)
- `~/.ssh/config`: the file for users to specify a port for SSH, add this to change it to 443:
```Host github.com Hostname ssh.github.com Port 443 User git```

## Git Configuration
- `git`: command that lets the command line know that what comes after is a git subcommand
- `git init`: set up the git system in the current directory (command line) to initalize a git repo
- `~/.gitconfig`: file used to store all the git config settings below
- `git config --global`: change the default of new projects on this user account (of the computer) only
  - `--system` is all users on the computer and their repos, `--global` is this user and its repos, `--local` (default) is only the current repo
- `git config --global user.name <user_name>`: set the default name for the user on commits to be `<user_name>`
- `git config --global user.email <user_email>`: set the default email for the user on commits to be `<user_email>`
- `git config --global init.defaultBranch main`: set the default name of new git projects to be `main`
- (MacOS) `echo .DS_Store >> ~/.gitignore_global`: write `.DS_Store` (file created by Finder) to `~/.gitignore_global`
- (Windows) `echo Thumbs.db >> ~/.gitignore_global`: write `.Thumbs.db` (file created by Windows Explorer) to `~/.gitignore_global`
  - `~/.gitignore_global` is just a hidden list of user defined files, it needs to be included in the git config file to do anything
- `git config --global core.excludesfile ~/.gitignore_global`: exclude the `~/.gitignore_global` file from being tracked
- `git config --global core.editor "code --wait"`: sets the default text editor to VS Code
- `git remote`: list the aliases of the remote repos tracked by the local repo, `origin` is typically one of them
- `git remote -v`: list the aliases and URLs of the remotes repo tracked by the local repo, `-v` for `--verbose`
- `git remote add <rrepo_alias> <rrepo_url>`: start tracking `<rrepo_url>` called `<rrepo_alias>` (again, `origin` is typically one of them) as a remote repo
- `git remote remove <rrepo_alias>`: stop tracking `<rrepo_alias>` as a remote repo

## Navigation and Branch Management
- `git switch <commit_id>`: switch `HEAD` to a specific `<commit_id>` or tip of a branch if a branch name is given
- `git switch -c <branch_name>`: creates a branch called `<branch_name>` and switches `HEAD` to it
  - edited and staged files follow to the new branch
- `git branch`: show list of local branches
- `git branch -r`: show list of remote branches, `-r` for `--remote`
- `git branch --all`: show list of all branches local and remote
- `git branch <branch_name>`: creates a branch called `<branch_name>`, but don't switch to it
- `git branch -d <branch_name>`: deletes `<branch_name>`, `-d` for `--delete`
- `git branch -D <branch_name>`: deletes `<branch_name>` even if it has unmerged changes, `-D` for `--delete --force`
- `git branch -u <rrepo_alias>/<remote_branch>`: set the default `<rrepo_alias>` and `<remote_branch>` for the current branch
- `git branch -u <rrepo_alias>/<remote_branch> <branch_name>`: set the default `<rrepo_alias>` and `<remote_branch>` for `<branch_name>`
  - `-u` for `--set-upstream-to=<repo_alias>/<remote_branch>` (note: the syntax requires a `\` unlike when `-u` used with `git push`)
- `git revert <branch_id>`: open text editor for commit message, save and exit to commit reverting changes made in `<branch_id>` or in range `<branch_id>...<branch_id>`
- `git revert <branch_id> -m "type: description"`: commit reverting changes made in `<branch_id>` with the message "type: description"
- `git reset <commit_id`: move HEAD to `HEAD~n` or `<commit_id>` and unstage (but keep) changes made up to HEAD before moving, this is the same as using `--mixed` (default behavior)
  - `git reset --soft <commit_id>`: move HEAD to `<commit_id>` and stage changes made up to HEAD before the reset
  - `git reset --hard <commit_id>`: move HEAD to `<commit_id>` and discard changes made after `<commit_id>`
  - if you mess up, use `git reflog` to find the `<commit_id>` or `HEAD@{n}` pointer needed for a `git reset --hard`

## Workflow
- `git add <file>`: add `<file>` that has been edited to the staging area (and start tracking it if not already)
- `git rm --cached <file>`: stop tracking `<file>`
- `git rm <file>`: delete `<file>`
- `git restore <file>`: restore `<file>` to its original commit state (undo edits)
- `git restore --staged <file>`: remove `<file>` from the staging area (keep edits)
- `git restore --source=<commit_id> <file>`: restore `<file>` from `<commit_id>` (edits, is unstaged)
- `git commit`: open text editor for commit message, save and exit to commit staged files
- `git commit -a`: open text editor for commit message, save and exit to commit stage files after automatically staging tracked files
- `git commit --amend`: replace the commit with a new commit (new commit ID as well)
- `git commit -m "type: description"`: commit staged files with the message "type: description"
- `git diff <commit_id_0> <commit_id_1>`: show difference between `<commit_id_0>` and `<commit_id_1>`
- `git push`: upload the current branch's new commits to the default remote repo and branch
- `git push <rrepo_alias> <remote_branch>`: upload the current branch's new commits to `<rrepo_alias>`'s `<remote_branch>`
- `git push -u <rrepo_alias> <remote_branch>`: set the current branch's default remote repo and remote branch to track `<rrepo_alias>` and `<remote_branch>` before uploading, `-u` for `--set-upstream-to=<repo_alias>/<remote_branch>`
- `git push --force-with-lease`: update the remote repo overwriting existing commit
- `git fetch`: download all commits on the default remote repo, but don't merge with local repo
- `git fetch <rrepo_alias>`: download all commits on `<rrepo_alias>`, but don't merge
- `git fetch <rrepo_alias> <remote_branch>`: download only the commits on `<remote_branch>` of `<rrepo_alias>`, but don't merge
- `git pull`: download all commits on the default remote repo and merge with local repo (`--merge` is the default behavior unless changed in `~/.gitconfig`, add `--rebase` to rebase instead)
- `git pull <rrepo_alias>`: download all the commits on `<rrepo_alias>` and merge
- `git pull <rrepo_alias> <remote_branch>`: download only the commits on `<remote_branch>` of `<rrepo_alias>` and merge
- `git merge <rrepo_alias>/<remote_branch>`: if fetched, merge `<remote_branch>` of `<rrepo_alias>` into current branch
  - merging, and merging by pulling, creates a new commit record

## Advanced Workflow
- (interactive staging) `git add -p`: stage chunks of changes at a time that can be committed separately, `-p` for `--patch`
  - `y` to stage, `n` to skip, `s` to split, `e` to edit, `q` to stop, and `git commit`, repeat until done
- (rebasing) `git rebase <rrepo_alias>/<remote_branch>`: if fetched, start rebasing the current branch to `<remote_branch>` of `<rrepo_alias>`
  - if a conflict is found, edit the file and `git add file.txt` then `git rebase --continue` to continue, `git rebase --abort` to stop or `git rebase --skip` to drop current commit from the new set of commits
- (squashing) `git rebase --interactive <commit_id>`: start interactive rebase onto `<commit_id>`
  - select action for commits between HEAD and `<commit_id>`, `git add file.txt` then `git rebase --continue` to continue, same as standard rebase
- `git cherry-pick <commit_id>`: merge `<commit_id>` onto the current branch

## Stashes
- `git stash`: create a temporary branch that snapshots the current working tree and staging area
- `git stash save "type: description"`: create a temporary branch that snapshots the current working tree and staging area with the message "type: description"
- `git stash list`: list stashes
- `git stash apply stash@{n}`: applies state of the working tree and staging area of the nth stash
- `git stash pop stash@{n}`: applies the nth stash and removes it
- `git stash drop stash@{n}`: removes the nth stash
- `git stash clear`: removes all stashes

## History
- `git status` show changes since previous commit
- `git log` show commit history for the current branch, and the flags below can be combined
- `git log --all` show commit history of all branches
- `git log --graph` show commit tree of the current repo
- `git log --oneline` show commit info on a single line per commit
- `git reflog` show log of pointer actions made on the current repo and their commit reference IDs ordered by most recent

## Commit Rules
- atomic commits
  - one commit per logical change, no touching anything like syling or refactoring if it doesn't need to be touched for the task at hand
  - keeps history easy to read, review, and revert if needed 
  - can lead to enormous commit history unless branching from, and squashing back into, main

`<type>: <subject>`

or
```
<type>: <subject>
<BLANK LINE>
<body>
<BLANK LINE>
<footer>
```
- messages should:
  - be no more than 100 characters wide
  - start with a `<type>`:
    - build: updating tools and libraries
    - chore: updating dependencies
    - ci: updating automation
    - docs: updating documentation, no code
    - feat: updating features
    - fix: fixing bugs
    - perf: performance improvements
    - refactor: cleaning up code
    - revert: `<original commit message>`
    - style: formatting code
    - test: updating tests

- the `<subject>` should:
  - be in the inperative mood, like a command or instruction
  - not be capitalized (should be read: "With this commit, ***add dark mode ui***")
  - not end with a period

- the `<body>` should:
  - be in the present tense
  - if the commit is a revert, it should be "This reverts commit `<hash>`."

- the `<footer>` should:
  - be used if a commit introduces code that could break existing code
  - start with "BREAKING CHANGE:" and explain what might make code no longer backwards compatible
  - if there is such a code change, message should start with `<type>!` instead of just `<type>`
